# f2b-manager 模块拆分与执行方案

> 目标：将项目拆分为可并行执行的独立模块，明确功能边界、接口契约、依赖关系与难度评级，以便分发到不同窗口分别执行。

## 二、各模块详细定义

### M0 — 基础设施层（Foundation）

| 项 | 内容 |
|----|------|
| **功能边界** | 项目骨架、配置加载、数据模型定义、SQLite 状态库、通用工具（shell/logger/distro）。**定义所有跨模块共享的接口契约与数据类型**，是其他模块的依赖基石。 |
| **产出文件** | `pyproject.toml`、`requirements.txt`、`f2b_manager/__init__.py`、`__main__.py`、`cli.py`、`config.py`、`app.py`（骨架）、`utils/{shell,distro,logger}.py`、`storage/{database,models}.py`、`config/config.example.yaml` |
| **输入** | 设计方案.md |
| **输出（接口契约）** | 见下方「接口契约」节——定义所有 dataclass、Protocol、DB schema |
| **难度** | ⭐⭐ — 代码量不大但需一次性定义好所有共享类型，是后续并行开发的地基 |
| **推荐模型** | **DeepSeek-V4-Pro**（系数 0.16） |
| **选择理由** | 契约定义需代码准确性但不需创造性推理，dataclass/Protocol/SQL schema 模式固定。DeepSeek 代码能力扎实，系数仅 GLM 的 1/6.6，是基础设施层的最优性价比选择 |

**M0 必须产出的接口契约（其他模块的开发依据）：**

```python
# storage/models.py — 全局数据模型
@dataclass
class Fail2banStatus: version: str; running: bool; jail_count: int; total_bans: int
@dataclass
class JailInfo: name: str; enabled: bool; current_ban: int; total_failed: int
@dataclass
class JailStatus(JailInfo): banned_ips: list[str]; findtime: str; bantime: str
@dataclass
class BanEvent: ip: str; jail: str; action: str; failures: int; matches: str; timestamp: datetime
@dataclass
class InstallResult: success: bool; message: str; version: str; details: list[str]

# 跨模块 Protocol（定义能力契约，解除编译期依赖）
class IFail2banManager(Protocol):
    def get_status(self) -> Fail2banStatus: ...
    def get_banned_ips(self) -> list[str]: ...
    def ban_ip(self, ip: str, jail: str) -> bool: ...
    def unban_ip(self, ip: str) -> bool: ...

class IMessageSender(Protocol):
    async def send_alert(self, chat_id: int, message: str, parse_mode: str = "HTML") -> bool: ...
    async def send_report(self, chat_id: int, message: str) -> bool: ...

class IStateDB(Protocol):
    def record_ban(self, event: BanEvent) -> None: ...
    def get_current_bans(self) -> list[tuple[str, str]]: ...
    def get_ban_history(self, days: int) -> list[BanEvent]: ...
```

```sql
-- storage/database.py — SQLite schema（M0 落地，M3/M4 使用）
CREATE TABLE ban_history (id, ip, jail, action, failures, country, timestamp, notified);
CREATE TABLE current_bans (ip, jail, banned_at, PRIMARY KEY(ip, jail));
CREATE TABLE daily_stats (date, total_bans, unique_ips, top_country);
CREATE TABLE config_overrides (key, value);
```

### M1 — Fail2ban 管理模块

| 项 | 内容 |
|----|------|
| **功能边界** | fail2ban 的安装/卸载/更新、运行时状态查询、手动封禁/解封、jail 配置生成、fail2ban-client 输出解析 |
| **产出文件** | `f2b_manager/fail2ban/{installer,manager,config_builder,parser}.py`、`config/jail.local.template` |
| **输入** | M0 的 `utils/shell.py`（subprocess 封装）、`utils/distro.py`（发行版检测）、`storage/models.py`（返回类型） |
| **输出** | 实现 `IFail2banManager` 接口 + `Fail2banInstaller` 类，供 M2/M4 调用 |
| **难度** | ⭐⭐⭐⭐ — 多发行版适配（apt/dnf/yum/apk/pacman）、subprocess 错误处理、fail2ban-client 输出格式解析（不同版本格式有差异）、配置文件模板生成 |
| **推荐模型** | **DeepSeek-V4-Pro**（系数 0.16） |
| **选择理由** | 涉及系统级操作和跨发行版兼容，但核心是 subprocess 调用 + 字符串解析 + if/else 分支，代码模式清晰。DeepSeek 代码逻辑能力属第一梯队，系数仅 0.16，性价比远超 GLM（1.06） |

**关键难点：**
- `fail2ban-client status` 输出解析需兼容 0.11.x 和 1.0.x 两种格式
- 安装流程需处理：已安装检测、依赖缺失、服务启动失败等异常路径
- `jail.local` 生成需动态拼接 telegram-notify action（与 M3 的 config 联动）

### M2 — Telegram Bot 模块

| 项 | 内容 |
|----|------|
| **功能边界** | Bot 生命周期管理、17 条命令处理、三级鉴权、危险操作二次确认（ConversationHandler）、长耗时操作异步进度通知、消息格式化 |
| **产出文件** | `f2b_manager/telegram_bot/{bot,auth,keyboards,formatters}.py`、`handlers/{status,manage,ban,report,config}.py` |
| **输入** | M0 的 `config.py`（bot_token/chat_ids）、`IFail2banManager`（M1，开发期用 mock）、`IMessageSender` 协议 |
| **输出** | `F2BTelegramBot` 类，实现 `IMessageSender`，供 M3/M4 调用发送消息 |
| **难度** | ⭐⭐⭐⭐⭐ — 全项目最复杂：python-telegram-bot v21 异步 API、17 个命令处理器、ConversationHandler 状态机、鉴权装饰器、内联键盘交互、异步任务进度反馈 |
| **推荐模型** | **GLM-5.2**（系数 1.06） |
| **选择理由** | 全项目唯一使用旗舰模型的模块。代码量最大（~900 行）、异步并发+状态机+框架深度叠加，是项目成败关键。GLM 中文技术理解最强且原生适配 WorkBuddy 调度架构，Token 利用率优于名义系数。一次写对省下的返工成本远超与 DeepSeek 的 6.6 倍模型差价 |

**关键难点：**
- `/uninstall` 需 ConversationHandler 三态（确认→执行→结果），中途可取消
- `/install` `/update` 耗时长，需异步执行 + 实时推送进度（"正在安装依赖...""正在生成配置..."）
- 鉴权装饰器需兼容同步/异步 handler
- 17 条命令的 help 文本、错误处理、参数校验量大

### M3 — 实时预警模块

| 项 | 内容 |
|----|------|
| **功能边界** | 接收 fail2ban action hook 事件、构造预警消息、IP 归属地查询、消息去重限流、调用 Bot 发送 |
| **产出文件** | `f2b_manager/notify/{sender,geoip,dedup}.py`、`scripts/f2b-notify.sh`、`config/telegram-notify.conf` |
| **输入** | M0 的 `config.py`、`storage/database.py`（记录事件）、`IMessageSender`（M2 Bot，开发期用 mock） |
| **输出** | `AlertSender` 类 + CLI `notify` 子命令（供 notify.sh 调用） |
| **难度** | ⭐⭐⭐ — 消息模板拼接直观，难点在去重逻辑（同 IP 时间窗口聚合）和 GeoIP 本地库/API 双模式 |
| **推荐模型** | **DeepSeek-V4-Pro**（系数 0.16） |
| **选择理由** | 逻辑中等复杂，模式清晰（接收→查询→构造→发送），去重窗口和 GeoIP 双模式回退 DeepSeek 可胜任。CLI→守护进程的 Unix Socket 桥接有一定技巧性，但 DeepSeek 代码能力足够 |

**关键难点：**
- `notify.sh` → CLI → 守护进程的桥接：CLI 是独立进程，需通过 Unix Socket/HTTP 把事件投递给常驻守护进程
- 去重窗口：同 IP 5 分钟内多次 ban 只通知一次，但状态库要记录全部
- GeoIP 本地 mmdb 查询 + 在线 API 回退

### M4 — 定时任务与监控模块

| 项 | 内容 |
|----|------|
| **功能边界** | APScheduler 任务调度、每日/每周报告生成、轮询兜底（对比状态库补发遗漏事件）、fail2ban 健康检查 + 自动恢复 |
| **产出文件** | `f2b_manager/monitor/{scheduler,reporter,health}.py` |
| **输入** | M0（config/storage）、M1（`IFail2banManager` 查状态）、M2（`IMessageSender` 发报告）、M3（`AlertSender` 补发预警） |
| **输出** | `F2BScheduler` 类，在 `app.py` 中与 Bot 共享事件循环启动 |
| **难度** | ⭐⭐⭐ — APScheduler 用法标准，难点在轮询 diff 算法（对比 current_bans 表）和报告数据聚合（SQL 统计 + Top 排名） |
| **推荐模型** | **DeepSeek-V4-Pro**（系数 0.16） |
| **选择理由** | 集成层模块，接口已由 M0 定义清楚，核心是 APScheduler 标准用法 + SQL 聚合 + 轮询 diff 算法。不需创造性推理，DeepSeek 代码+逻辑能力足够处理异步任务编排 |

**关键难点：**
- APScheduler 的 `AsyncIOScheduler` 必须与 python-telegram-bot 的 `Application` 共享同一事件循环
- 轮询 diff：`fail2ban-client banned` 结果 vs `current_bans` 表，找出 added/removed
- 健康检查失败时的自动重启逻辑（systemctl restart + 通知 + 重试上限）

## 三、接口契约汇总（并行开发的关键）

各模块并行开发的前提是**严格依赖 M0 定义的契约，而非具体实现**。开发期用 mock/stub 替代未完成的依赖：

```
M2 (Bot) 开发时:
  - 需要 IFail2banManager → 用 MockFail2banManager（返回固定 Fail2banStatus）
  - 需要 IStateDB         → 用 MockStateDB

M3 (Notify) 开发时:
  - 需要 IMessageSender   → 用 MockMessageSender（打印到日志）

M4 (Monitor) 开发时:
  - 需要 IFail2banManager + IMessageSender + AlertSender → 全部 mock
  - 本模块在 Wave 3 开发，此时 M1/M2/M3 已完成，可直接用真实实现
```

## 四、难度与模型匹配方法论

### 4.1 难度维度

| 维度 | 说明 | 权重 |
|------|------|------|
| 逻辑复杂度 | 异步/并发/状态机/算法 | 40% |
| 代码量 | 预估行数 | 20% |
| 领域知识 | Linux系统管理/框架API/安全 | 25% |
| 集成点 | 跨模块依赖数量 | 15% |

### 4.2 难度评级标准

| 评级 | 综合分 | 特征 | 模型能力要求 |
|------|--------|------|-------------|
| ⭐⭐ | 2-3 | 模板化、模式固定、少量决策 | 基础编码能力即可 |
| ⭐⭐⭐ | 3-4 | 有明确算法/逻辑，需处理边界条件 | 扎实的代码能力 + 领域基础 |
| ⭐⭐⭐⭐ | 4-5 | 多发行版适配/异步/系统集成，边界条件多 | 强代码能力 + 系统级知识 |
| ⭐⭐⭐⭐⭐ | 5+ | 大量异步并发 + 状态机 + 框架深度 + 安全 | 顶级推理 + 框架精通 + 严谨 |

### 4.3 模型匹配策略

> **原则**：在保证质量的前提下优先性价比。高难度模块用强模型保质量，低难度模块用轻量模型省成本。

| 难度 | 匹配模型层级 | 策略 |
|------|-------------|------|
| ⭐⭐⭐⭐⭐ | 旗舰/推理型模型 | 质量优先，M2 是项目成败关键 |
| ⭐⭐⭐⭐ | 旗舰/推理型模型 | 系统级操作不容出错 |
| ⭐⭐⭐ | 标准型模型 | 平衡性价比 |
| ⭐⭐ | 轻量型模型 | 成本优先 |

### 6.2 选型逻辑详解

**为什么只有 M2 用 GLM-5.2，其余用 DeepSeek-V4-Pro？**

M2 是整个项目代码量最大（~900 行）、复杂度最高（⭐⭐⭐⭐⭐）的模块：
- python-telegram-bot v21 异步 API 细节多，回调签名、事件循环共享容易出错
- 17 个命令处理器 + ConversationHandler 状态机，交互逻辑密集
- 鉴权装饰器需兼容同步/异步 handler，有 Python 元编程陷阱
- 一旦写错，返工成本远超模型差价（GLM 系数 1.06 vs DeepSeek 0.16，差 6.6 倍，但 M2 只占 7 个模块中的 1 个）

其余 5 个代码模块（M0/M1/M3/M4/M6）的难度驱动因素是"代码准确性+边界条件处理"，DeepSeek-V4-Pro 的代码能力在国产模型中属第一梯队，系数仅 0.16，性价比碾压。

**为什么 M5 用 Hy3 而非 DeepSeek？**

M5 产出是 Shell 脚本 + systemd unit + Markdown 文档，设计方案中已有完整 `install.sh` 骨架。本质是"照模板填实"，不需要代码推理能力。Hy3 系数 0.04（DeepSeek-V4-Pro 的 1/4，GLM 的 1/26），写 Shell 和文档绰绰有余。
