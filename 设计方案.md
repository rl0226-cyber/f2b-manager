# VPS Fail2ban 管理系统 - 详细设计方案

> 项目代号：**f2b-manager**
> 目标：在 VPS 上运行的一体化 fail2ban 管理程序，集成安装/卸载/更新、Telegram 机器人定时报告与实时预警。

---

## 一、需求拆解

| 编号 | 需求 | 说明 |
|------|------|------|
| R1 | fail2ban 安装 | 自动识别发行版，安装 fail2ban 并完成初始配置 |
| R2 | fail2ban 卸载 | 安全卸载，保留配置备份 |
| R3 | fail2ban 更新 | 通过包管理器或源码更新 |
| R4 | Telegram 机器人 | 连接 Bot，提供交互命令 |
| R5 | 定时任务 | 定时推送 IP 封禁情况汇总报告 |
| R6 | 实时预警 | 当 fail2ban 封禁/解封 IP 时，实时推送通知到 Telegram |

---

## 二、技术选型

### 2.1 开发语言与核心依赖

| 组件 | 选型 | 理由 |
|------|------|------|
| 开发语言 | **Python 3.10+** | 系统管理 + 网络请求生态成熟，VPS 默认自带 |
| Telegram SDK | **python-telegram-bot (v21+)** | 官方推荐，异步支持好，社区活跃 |
| 定时任务 | **APScheduler** | 轻量，无需额外组件（cron 也可，但 APScheduler 更灵活） |
| 配置管理 | **PyYAML** | 可读性强，配置即文档 |
| 进程管理 | **subprocess** | 标准库，调用 fail2ban-client / 包管理器 |
| 服务管理 | **systemd** | 主流发行版标配，开机自启 |
| HTTP 请求 | **httpx**（异步） | python-telegram-bot 已内置依赖 |

### 2.2 运行环境要求

- Linux 发行版：Debian/Ubuntu、CentOS/RHEL/Rocky、Alpine（可选）
- Python ≥ 3.10
- systemd
- root 权限（安装/管理 fail2ban 必需）
- 出站网络访问 `api.telegram.org`

### 3.2 实时预警数据流（关键设计）

实时预警采用 **Hook + 轮询兜底** 双保险机制：

```
[攻击发生]
    │
    ▼
[fail2ban filter 检测到匹配日志]
    │
    ▼
[fail2ban 执行 ban action]
    │
    ├──► [默认 action: iptables 封禁 IP]
    │
    └──► [自定义 action: telegram-notify]
              │
              ▼
         notify.sh (shell)
              │
              ▼
    调用 f2b-manager CLI: f2b-notify ban <ip> <jail> <failures>
              │
              ▼
    f2b-manager 通过 Telegram Bot API 发送预警消息
              │
              ▼
         用户 Telegram 收到实时通知

[兜底] Scheduler 每 N 分钟轮询 fail2ban-client status
    │   对比状态库，发现遗漏的 ban/unban 事件 → 补发通知
    ▼
```

#### 4.1.1 安装器 `installer.py`

```python
class Fail2banInstaller:
    """负责 fail2ban 的安装、卸载、更新"""

    def detect_distro(self) -> DistroInfo:
        """检测发行版，返回包管理器类型
        读取 /etc/os-release，识别:
        - debian/ubuntu → apt
        - centos/rhel/rocky/fedora → dnf/yum
        - alpine → apk
        - arch → pacman
        """

    def install(self) -> InstallResult:
        """安装流程:
        1. 检查是否已安装 (which fail2ban-server)
        2. 检测发行版
        3. apt-get update && apt-get install -y fail2ban
           或 dnf install -y fail2ban / yum install -y fail2ban
        4. 生成 jail.local (基于模板，含 Telegram action)
        5. 部署 telegram-notify action 配置
        6. 部署 notify.sh 脚本到 /usr/local/bin/
        7. systemctl enable --now fail2ban
        8. 验证: fail2ban-client ping
        9. 返回安装结果
        """

    def uninstall(self, keep_config: bool = True) -> UninstallResult:
        """卸载流程:
        1. systemctl stop fail2ban
        2. systemctl disable fail2ban
        3. apt-get remove --purge fail2ban / dnf remove fail2ban
        4. 若 keep_config: 备份 /etc/fail2ban 到 /etc/fail2ban.backup.YYYYMMDD
        5. 清理 notify.sh 和 action 配置
        6. 返回卸载结果
        """

    def update(self) -> UpdateResult:
        """更新流程:
        1. 记录当前版本 (fail2ban-client --version)
        2. apt-get update && apt-get install --only-upgrade fail2ban
           或 dnf upgrade fail2ban
        3. systemctl restart fail2ban
        4. 记录新版本，返回版本变化
        """
```

### 4.3 定时任务与监控模块 (`f2b_manager/monitor/`)

#### 4.3.1 调度器 `scheduler.py`

```python
class F2BScheduler:
    """基于 APScheduler 的定时任务"""

    def __init__(self, config, reporter, f2b_manager, bot):
        self.scheduler = AsyncIOScheduler()

    def setup_jobs(self):
        # 任务1: 每日封禁报告
        self.scheduler.add_job(
            self.reporter.daily_report,
            trigger='cron', hour=8, minute=0,
            id='daily_report'
        )

        # 任务2: 每周详细统计
        self.scheduler.add_job(
            self.reporter.weekly_report,
            trigger='cron', day_of_week='mon', hour=8,
            id='weekly_report'
        )

        # 任务3: 轮询兜底 (每 5 分钟)
        self.scheduler.add_job(
            self.monitor.poll_ban_changes,
            trigger='interval', minutes=5,
            id='poll_changes'
        )

        # 任务4: fail2ban 健康检查 (每 10 分钟)
        self.scheduler.add_job(
            self.monitor.health_check,
            trigger='interval', minutes=10,
            id='health_check'
        )
```

#### 4.3.3 轮询监控器 `monitor.py`

```python
class BanMonitor:
    """轮询兜底 + 状态追踪"""

    def __init__(self):
        self.state_db = StateDB()  # SQLite 轻量状态库

    def poll_ban_changes(self):
        """轮询检测封禁变化（兜底 hook 遗漏的事件）
        1. fail2ban-client banned → 获取当前封禁列表
        2. 与 state_db 中上次记录对比
        3. 新增的 IP → 补发 ban 通知
        4. 消失的 IP → 补发 unban 通知
        5. 更新 state_db"""

    def health_check(self):
        """fail2ban 健康检查
        - fail2ban-client ping
        - 若失败 → 发送告警: ⚠️ fail2ban 服务异常
        - 检查 systemd 服务状态
        - 若服务挂了 → 尝试自动重启 + 通知"""
```

# 服务停止时通知

actioncheck =

# 封禁 IP 时通知 ★关键

# 解封 IP 时通知
actionunban = /usr/local/bin/f2b-notify.sh "unban" "<ip>" "<name>"

actionstart_on_demand = false

#### 4.4.2 通知脚本 `notify.sh`

**文件: `/usr/local/bin/f2b-notify.sh`**

```bash
#!/bin/bash
# Fail2ban -> f2b-manager 通知桥接脚本
# 被 fail2ban action 调用，将事件转发给 f2b-manager

# 参数: $1=事件类型 $2=IP/jail $3=jail $4=failures $5=matches
EVENT="$1"

# 通过 f2b-manager CLI 转发（CLI 内部走 HTTP/Unix Socket 调用守护进程）
/usr/local/bin/f2b-manager notify \
    --event "$EVENT" \
    --ip "$2" \
    --jail "$3" \
    --failures "$4" \
    --matches "$5" \
    >/dev/null 2>&1 &

exit 0  # 永远返回 0，避免影响 fail2ban 正常工作
```

## 七、部署流程

### 7.1 一键安装 `install.sh`

```bash
#!/bin/bash
# f2b-manager 一键安装脚本

set -e

INSTALL_DIR="/opt/f2b-manager"
CONFIG_DIR="/etc/f2b-manager"
VENV_DIR="$INSTALL_DIR/venv"

echo "=== f2b-manager 安装程序 ==="

# 1. 检查 root 权限
[ "$(id -u)" -ne 0 ] && echo "请使用 root 运行" && exit 1

# 2. 检查 Python 版本
python3 -c "import sys; sys.exit(0 if sys.version_info >= (3,10) else 1)" \
    || { echo "需要 Python 3.10+"; exit 1; }

# 3. 创建目录
mkdir -p "$INSTALL_DIR" "$CONFIG_DIR"

# 4. 复制程序文件
cp -r f2b_manager/ "$INSTALL_DIR/"
cp pyproject.toml requirements.txt "$INSTALL_DIR/"

# 5. 创建虚拟环境 + 安装依赖
python3 -m venv "$VENV_DIR"
"$VENV_DIR/bin/pip" install --upgrade pip
"$VENV_DIR/bin/pip" install -r "$INSTALL_DIR/requirements.txt"

# 6. 部署配置
if [ ! -f "$CONFIG_DIR/config.yaml" ]; then
    cp config/config.example.yaml "$CONFIG_DIR/config.yaml"
    echo "⚠️  请编辑 $CONFIG_DIR/config.yaml 填入 bot_token"
fi
chmod 600 "$CONFIG_DIR/config.yaml"

# 7. 安装 fail2ban + 配置 action
# (调用 f2b-manager CLI 完成)
"$VENV_DIR/bin/python" -m f2b_manager fail2ban install

# 8. 部署 systemd 服务
cp systemd/f2b-manager.service /etc/systemd/system/
systemctl daemon-reload
systemctl enable f2b-manager

# 9. 部署 notify 脚本
cp scripts/f2b-notify.sh /usr/local/bin/
chmod +x /usr/local/bin/f2b-notify.sh
ln -sf "$VENV_DIR/bin/python" /usr/local/bin/f2b-manager-python

echo "✅ 安装完成！"
echo "1. 编辑配置: vim $CONFIG_DIR/config.yaml"
echo "2. 启动服务: systemctl start f2b-manager"
echo "3. 查看日志: journalctl -u f2b-manager -f"
```

## 八、安全设计

### 8.1 权限控制

| 风险 | 缓解措施 |
|------|----------|
| Bot 被盗用执行危险命令 | chat_id 白名单，三级权限，危险操作二次确认 |
| 配置文件泄露 token | 文件权限 600，仅 root 可读 |
| fail2ban action 脚本被篡改 | notify.sh 固定路径 + root 所有 + 755 权限 |
| 消息刷屏 | 限流 + 去重窗口（同 IP 5 分钟内只通知一次） |
| 轮询暴露敏感信息 | 状态库权限 600，日志脱敏 |

### 8.2 网络安全

- Telegram Bot 使用 HTTPS（TLS 1.2+）
- webhook 模式建议配置反向代理 + TLS
- 不开放额外端口（polling 模式仅需出站 443）
- 状态库 / 配置文件不对外暴露

## 十一、关键设计决策说明

### 11.1 为什么用 Hook + 轮询双保险？

- **Hook（action）**: 实时性最好，fail2ban 执行 ban 时立即触发。但若 notify.sh 执行失败（网络问题、f2b-manager 重启中），事件会丢失。
- **轮询兜底**: 每 5 分钟对比状态，补发遗漏事件。代价是延迟，但保证最终一致。
- **组合**: Hook 负责"快"，轮询负责"全"。

### 11.2 为什么用独立守护进程而非 cron？

- Bot 需要常驻接收消息（polling），必须守护进程
- 守护进程内集成 APScheduler，避免 cron + 脚本碎片化
- 状态库（SQLite）需要常驻连接，守护进程更高效
- systemd 管理生命周期，崩溃自动重启

### 11.3 为什么 notify.sh 通过 CLI 调用而非直接调 API？

- 解耦：notify.sh 不需要知道 bot_token（安全）
- 统一：所有通知走 f2b-manager，便于去重、限流、记录
- 健壮：f2b-manager 可缓存消息，网络恢复后重试

### 11.4 为什么用 SQLite 而非文件？

- 封禁历史需要查询、统计、去重，关系型更合适
- SQLite 零配置、单文件、性能足够（单机场景）
- 支持 WAL 模式，读写并发无忧
